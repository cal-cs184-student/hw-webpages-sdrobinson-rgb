Section I: Bezier Curves and Surfaces

Part 1: Bezier curves with 1D de Casteljau subdivision

For Part 1, I implemented the function evaluateStep, which evaluates a single level of recursion in Casteljau's algorithm. This is used by the GUI to visualize each recursive step of how the algorithm evaluates a point on a Bezier curve.

The algorithm works by taking the \(n\) control points of a Bezier curve and performing a linear interpolation with some value \(t\) in the range \([0, 1]\) between adjacent control points to get \(n - 1\) linearly interpolated points. This process is then performed recursively for each new set of points until only one point remains, and the resulting point will be a point on the Bezier curve. Casteljau's algorithm can be used to evaluate Bezier curves since various \(t\) values give different points along the curve, meaning any point on a Bezier curve can be recovered with Casteljau's algorithm given an appropriate value for \(t\).

For any given execution of Casteljau's algorithm on \(n\) control points, \(n - 1\) sets of linear interpolations must be completed. For the following example, the curve is defined by 6 control points, meaning 5 calls to evaluateStep can define a point.

Altering the control points of a curve will change the curve. Altering \(t\) does not affect the curve, but will change the point given by performing Casteljau's algorithm, essentially 'sliding' it along the resulting Bezier curve.

Part 2: Bezier surfaces with separable 1D de Casteljau

Casteljau's algorithm for evaluating Bezier curves can be extended to Bezier surfaces by performing the algorithm first in one dimension and using the calculated result to extend it into the second dimension of the surface. That is, for a 2D grid of points, individual rows of those points can be treated as 1D Bezier curves for which we can evaluate a point some \(t\) along the curve using Casteljau's algorithm. Then, given a series of evaluated points for each of these rows, we can define a perpendicular 1D Bezier curve, which we can again use Casteljau's algorithm on. Thus, for any two values of \(t\), defined as \((u, v)\), this method can evaluate the respective point on the Bezier surface.

My implementation of this process uses three functions, evaluateStep, which calculates a single set of linear interpolations for a single Bezier curve of 3D points, evaluate1D, which uses a series of evaluateStep calls to evaluate a point on a single 'row' of control points, as described above, and evaluate, which uses a series of evaluate1D calls to define a perpendicular Bezier curve, and one final evaluate1D call to get a single point on the Bezier surface.

Section II: Triangle Meshes and Half-Edge Data Structure

Part 3: Area-weighted vertex normals

Part 3 required me to implement Vertex::normal(), a member function of Vertex that calculated an area-weighted normal for this vertex. That is, a normal vector calculated by a weighted average of the adjacent faces' normal vectors, weighted by the faces' areas.

To implement this, I first evaluated two parallel vectors, one for the face normals and one for their areas, populating the vectors for each face adjacent to the current vertex. To calculate the normal of a face, I took the cross product of two vectors formed by the points defining the boundaries of the face, which gave a vector perpendicular to the face, and then normalized this vector to get the face normal. To find the area, I used Heron's formula, calculating the respective side lengths of the triangular face using the magnitude of the vector formed by subtracting sets of the bounding points.

From these two parallel vectors, I calculated the vertex normal by first summing the area vector, allowing me to easily calculate the weight of a given face by dividing its area by this sum. My implementation then multiplies this weight by the corresponding normal in the parallel vector and adds it to the vertex normal. The sum of all these values, normalized, gives the normal vector for this vertex.

One use of vertex normals is in Phong shading.

Part 4: Edge flip

To implement flipEdge for Part 4, I first confirmed that I wouldn't break any vertex or face half-edge reference by assigning the half-edge of the vertices connecting the edge to unchanging half edges and assigning the half edge of each face to the half edge of the edge I would be flipping. From there, everything I had to reassign was on all of the half edges of the mesh, since none of the half edges corresponding to any of the other mesh objects would be changing (beyond changing their properties, that is).

I reassigned every pointer in the two flipping half-edges using the provided setNeighbors function. For the bounding half-edges of the mesh, I only reassigned next and face since the edge, vertex, and twin of each bounding half edge were not impacted by the edge being flipped and thus would not need to be altered.

One thing I had to consider when implementing this function was the order in which I would change the next pointer on the half-edges. I had to be sure to order them in a way such that I wouldn't 'lose' any half-edge by reassigning the only pointer that I could reach it from. I leveraged the fact that I had direct references to the two half-edges being flipped (given by the half-edge pointer on the provided edge and its twin). First I reassigned the next pointer of half-edge->next()->next() for both flipping edges. That is, the last half-edge of the face. This gave me a reference to the half-edge immediately following the flipping half edges, allowing me to reassign those pointers without losing access to any half-edges. Since the next pointer of those half-edges then goes to the previously third edge, I can access the final set of edges by using half-edge->next()->next() again--as now this series of pointers references a different half-edge--to reassign the final set of pointers.

Part 6: Loop subdivision for mesh upsampling

In Part 6, I used both splitEdge and flipEdge to implement loop subdivision. The process of loop subdivision is to first split every edge in the mesh, which I implemented by looping through every edge (that did not connect to a new vertex, to avoid an infinite loop). Then, any new edge that connects a new and old vertex should be flipped. To keep track of whether a vertex was new or old, I amended my implementation of splitEdge to update the isNew flag in the new vertex and edges.

In addition to adding new triangles, for proper loop subdivision, the positions of all new and old vertices have to be updated based on a specific calculation between the vertex's position (if it had one) and the positions of the surrounding nodes. As was recommended, I made all of these calculations prior to any of the edge splitting and flipping described above, keeping the new position of the vertices in the provided newPosition variable vertices and edges had. Then, after I divided the mesh with splitEdge and flipEdge, I updated the positions of the vertices to these pre-calculated values. This is the most effective way to implement this calculation since the positions of all vertices need to be based on values from the original mesh, so calculating the positions as a mesh is split will lead to improper results. Finally, while updating the vertex positions, I made sure to clear the isNew flags from all vertices and edges so the mesh could be subdivided repeatedly without bugs.

Performing subdivision on meshes with sharp edges will gradually round out the sharp edges. The effect can be reduced by pre-splitting any sharp edges you want to preserve in a mesh. This works because loop subdivision calculates the position of a vertex based on the surrounding vertices. Pre-splitting an edge adds an additional vertex to that edge, which increases how much that edge 'contributes' to the upsampled mesh.
